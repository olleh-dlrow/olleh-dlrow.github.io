UE4 C++代码规范

什么是代码规范？

特定组织针对某一程序设计语言，根据语言特性、开发环境、运行环境等长期编程经验积累总结出来的一套适用于大部分场景的最佳编程实践。

为什么要遵守规范？

增加代码规范

增强可维护性

提高编码效率

降低代码复杂度

避免和减少常规bug

减少安全漏洞

提高代码编写水平

命名规范

每个单词首字母要大写 Upper Camel Case

类型名称前面要有一个标识类型的前缀

T：模板类

U：继承自UObject

A：继承自AACtor

S：继承自SWidget

I：继承自接口类

E：枚举类

b：布尔变量

F：其他类

类型和变量名一般是名词

函数或者方法名用动词来描述

返回值类型为布尔值的函数，使用Is或者Should为开头定义

基本数据类型

bool

TCHAR

int8/uint8

int16..

int32..

int64..

float

double

PTRINT：与指针同样大小的整数

字符串类

FString，FText，FName，TCHAR

容器类

TArray，TMap，TSet

文件相关操作

FFileManagerGeneric

序列化

FArchive

注释

编写有意义的注释

遵守JavaDoc规则

代码格式

大括号另起一行

if else另起一行

Switch case另起一行

如果多个case条件都执行相同的代码块，则添加注释//Fall through

Tab设置为Tab键，大小设置为4个字符

命名空间：

不要再全局作用域内使用using声明，可以在一个命名空间或者函数体内声明

文件依赖

所有头文件都要使用#pragma避免多次包含

可以使用前向声明Forward Declarations

谨慎使用FORCEINLINE（inline），这会导致强制重新编译

模块通常包括Public和Private目录，所有可能被其他模块用到的定义和声明，都应该位于Public目录，剩下的放到Private目录

封装

必须使用访问修饰符

C++11与现代语言语法

static_assert：编译时断言

static_assert(sizeof(TCHAR) == 4 || sizeof(TCHAR) == 2)

判断代码在哪个平台执行

auto关键字使用：

将Lambda表达式和变量绑定的时候使用

用作迭代器变量的时候可以使用

在模板开发中使用，尤其是当表达式的类型不太容易辨别的时候

Lambda表达式长度不要过长

以引用或者值传递变量时，如果Lambda不再被捕获的变量上下文执行，那么可能导致无效引用

使用值传递捕捉时，要避免不必要拷贝

对于规模较大的Lambda，要声明返回值

如果引用了成员变量，则会隐式包含this指针

枚举：

使用Enum Class代替旧的

UPROPERTY类型定义：

只要是unit8,使用TEnumAsByte<>定义

ENUM_Class_FLAGS(EnumType)自动定义位操作

使用using定义类型别名的声明

默认成员初始化

缺点：

可能导致重新编译

头文件修改无法在Patch版本做到

有些对象无法这样初始化：基类，UObject，前向声明指针等

第三方代码：

使用容易搜索的注释标签

@third party code

一般风格：

类的构造函数中初始化成员变量的顺序最好与类中声明的成员变量的顺序一致，否则可能导致编译器警告

减少变量的依赖距离，即把变量声明和定义放在第一次使用的地方，可以在代码段的最前面初始化变量，但不要等到几百行之后再使用

在文件结尾留一行，兼容gcc编译器

尽量解决所有的编译器警告

不要提交调试代码，将调试代码和其他代码混在一起会让可读性变差

确保对字符串变量使用TEXT（）宏，如果没有使用，那么对于从字符串构造FString的代码会产生不必要的转码处理

指针和引用应当只有一个空格，位于符号右边

禁止使用Shadowed Variable，即临时变量把成员变量覆盖了

不要硬编码字符串，应该在一个地方统一定义

如果希望一直持有UObject对象，必须添加UPROPERTY标记，防止被GC回收

对于字符编码转换系列宏，必须只能用作函数参数使用，不能将结果赋值给其他对象

错误：const char* p = TCHAR_TO_ANSI(SomeUnicodeString)
